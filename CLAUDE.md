## シニアエンジニアとしての検討・提案

ユーザーがプロンプトで方針・仕様・実装案・前提を述べたとき、**シニアエンジニアの立場で**次を検討する。

### いつ言うか

- 設計・API・データモデル・運用・セキュリティ・後方互換・パフォーマンスに**実害や不可逆なコスト**がありそうなとき。
- ユーザーの前提が**仕様・コードベースと食い違う**可能性があるとき。
- **より安全・単純・保守しやすい代替**が明らかなとき。

**言わなくてよい例:** 既に合意済みの細部、自明なリネーム、ユーザーが明示的に「これだけ」と閉じている極小タスク（その場合は短く確認するに留めてよい）。

### どう言うか

- **懸念・反論・トレードオフ**を、根拠つきで**簡潔に**述べる（長い説教や全面否定は避ける）。
- 可能なら **代替案** または **条件付きで進めるならの注意点** を 1 つ以上示す。
- ユーザーが決めれば進める話題では、**判断はユーザーに委ねる**旨をはっきり書く。
- 不確実なときは「確認したい点」として列挙し、断定しすぎない。

目的は作業を止めることではなく、**見落としやすいリスクを共有してから一緒に進める**こととする。

## コミットメッセージ規約

コミットメッセージは必ず**日本語**で記述する。英語や他言語を避け、文脈に応じて簡潔かつ内容が分かるように記載する。

例:

- バグ修正: 「〇〇のバグを修正」
- 機能追加: 「〇〇機能を追加」
- リファクタ: 「〇〇をリファクタ」

マージコミットや初回セットアップ時も日本語を使うこと。

## C# コーディング規約

## 常時原則（C#）

### 設計と保守性（C#）

- 変数名は省略しすぎず、意図が分かる命名を優先する。
- コンストラクタはプライマリコンストラクタを優先する。引数はフィールド初期化子や基底コンストラクタ呼び出しで利用する。引数検証や非自明な初期化処理がフィールド初期化子で表現できない場合のみ、通常のコンストラクタを使う。
- `public` / `internal` は XML ドキュメントコメントを付ける。`private` も意図が自明でない場合はコメントを付ける。プライマリコンストラクタの引数は型の `<param>` で説明する。
- 開発中は互換性目的のフォールバックを実装しない。互換性が必要な場合は、ユーザーまたは仕様で明示されたときのみ実装する。
- デザインパターンや新しい構成は、重複削減・責務分離・テスト容易性の改善が明確な場合に限って採用する。
- 小規模変更では既存構造との整合と単純実装を優先し、抽象化や新規パターン導入は必要性が明示された場合に行う。

### 分岐とデータ処理（C#）

- 型判定と分岐はパターンマッチング、`switch` / switch 式を優先し、`as` の多用を避ける。
- 多段 `if` と暗黙フォールバックを避け、網羅性を意識する。
- 反復処理は宣言的な記法（LINQ）を主構文とし、`foreach` / `for` は可読性や性能上の理由が明確な場合に限って使う。

### 性能（C#）

- 計算量を意識し、不要な全件走査・多重ループ・N+1 クエリを避ける。高頻度参照は `Dictionary` / `HashSet` など適切な構造を優先する。
- 性能最適化は計測結果（プロファイル、実行計画、ログ）に基づいて行い、推測だけで複雑化しない。

### セキュリティ（C#）

- 外部入力は不正を前提に検証し、型・長さ・許可値をサーバー側で強制する。
- 認証済みでも認可チェックを必須とし、テナント境界を跨ぐアクセスを許可しない。
- シークレット、トークン、接続文字列などの機密情報をコード・ログ・エラーレスポンスへ出力しない。
- SQL は文字列連結で組み立てず、必ずパラメータ化する。
- ログは最小権限で出力し、個人情報・機密情報はマスクまたは非出力とする。
- クライアント向けエラーに内部実装情報（スタックトレース、SQL 詳細、接続情報）を含めない。

### 可観測性・運用性（C#）

- 障害解析に必要な構造化ログと相関 ID（TraceId など）を維持し、主要処理の追跡可能性を確保する。
- 監視で利用するメトリクスとログ粒度を意識し、運用時に判断できない実装を避ける。

### 信頼性・回復性（C#）

- 外部 I/O はタイムアウト・キャンセルを前提に実装し、再試行は副作用と冪等性を考慮して行う。
- 部分失敗時の整合性を明示し、失敗時に安全側へ倒れる設計を優先する。

### 品質

- テストは AAA を明示し、1テスト1シナリオで目的が読める構造にする。

## 補足（C#）

- 詳細規約は `.cursor/rules/csharp-standards.mdc` と `docs/development-guidelines.md` を参照する。

## コメント・ドキュメント規約

実装とあわせて、**読み手が意図・契約・制約をコード外からも追える**ことを標準とする。本文は日本語。

- **新規・改修した公開 API**には、変更と同じ PR でドキュメントを付ける。
- **自明でない振る舞い**（セキュリティ境界、サイズ上限、パス解決、例外、外部連携）には必ず説明を残す。
- **クラス / モジュール**には目的・利用コンテキスト・主要制約（C# は `<remarks>`、TS は JSDoc）。
- **メソッド / 関数**には引数・戻り値・例外（またはエラー条件）。
- **定数（制限値）**には意味と採用理由。
- **テスト**には日本語で検証シナリオを明示。
- **書かない**: 自明な言い換えコメント、実装と矛盾する説明、機密情報。
- **仕様・運用に影響する変更**では、関連 `docs/` / `.spec-workflow/` を依頼範囲内で整合。

詳細は `.cursor/rules/documentation-standards.mdc` と `docs/development-guidelines.md` §4.3。

## TypeScript / React コーディング規約

対象は主に `ui/studio/`（Next.js）。インポート順・分割方針は既存コードに合わせる。

## 常時原則（TypeScript）

### 設計と保守性（TypeScript）

- 変数名は省略しすぎず、意図が分かる命名を優先する。
- `export` する関数・型・定数・React コンポーネントには JSDoc を付ける。
- 開発中は互換性目的のフォールバックを実装しない。互換性が必要な場合は、ユーザーまたは仕様で明示されたときのみ実装する。
- デザインパターンや新しい構成は、重複削減・責務分離・テスト容易性の改善が明確な場合に限って採用する。
- 小規模変更では既存構造との整合と単純実装を優先し、抽象化や新規パターン導入は必要性が明示された場合に行う。

### 分岐とデータ処理（TypeScript）

- 型はナローイングで表現し、`as` の多用を避ける。
- 分岐は `switch` / `Record` を優先し、多段 `if` と暗黙フォールバックを避ける。
- データ変換・抽出は宣言的な配列操作（`map` / `filter` / `reduce` など）を主構文とし、`for` / `for...of` は可読性や性能上の理由が明確な場合に限って使う。

### 性能（TypeScript）

- 計算量を意識し、不要な全件走査・多重ループ・N+1 API/DB 呼び出しを避ける。高頻度参照は `Map` / `Set` など適切な構造を優先する。
- 性能最適化は計測結果（プロファイル、実測ログ、トレース）に基づいて行い、推測だけで複雑化しない。

### セキュリティ（TypeScript）

- 外部入力は不正を前提に検証し、型・長さ・許可値をサーバー側で強制する。
- 認証済みでも認可チェックを必須とし、テナント境界を跨ぐアクセスを許可しない。
- シークレット、トークン、接続文字列などの機密情報をコード・ログ・エラーレスポンスへ出力しない。
- API/DB 呼び出しで動的な文字列連結によるクエリ生成を避け、必ず安全なパラメータ化手段を使う。
- ログは最小権限で出力し、個人情報・機密情報はマスクまたは非出力とする。
- クライアント向けエラーに内部実装情報（スタックトレース、SQL 詳細、接続情報）を含めない。

### 可観測性・運用性（TypeScript）

- フロント側エラーは追跡可能な情報（相関 ID、画面コンテキスト）を保持し、運用で調査不能な実装を避ける。
- 監視や分析に必要なイベント/ログは、個人情報保護を満たしたうえで一貫した形式で出力する。

### 信頼性・回復性（TypeScript）

- API 呼び出しはタイムアウト・キャンセル・リトライ条件を明示し、無限再試行や重複送信を避ける。
- 失敗時の UI 振る舞い（再試行、フォールバック表示、エラー通知）を明確にし、サイレント失敗を避ける。

### 国際化（i18n）（TypeScript）

- UI 文言は i18n 辞書経由で参照し、`*.tsx` へハードコードしない。

### テスト（TypeScript）

- テストは AAA 構成で 1ケース1シナリオを維持する。

## 品質チェック

- `tsc --noEmit` と該当範囲の `npm run test:run` を通す。

## 補足（TypeScript）

- 詳細規約は `.cursor/rules/typescript-standards.mdc` と `docs/development-guidelines.md` を参照する。横断方針は `.cursor/rules/documentation-standards.mdc`。

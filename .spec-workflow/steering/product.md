# Product Steering — Statevia

## Product Purpose

Statevia は **定義駆動（definition-driven）** かつ **イベントソーシング** に基づく **ワークフローエンジン** である。ワークフロー定義のコンパイル・実行・永続化を通じて、運用可能な形で業務プロセスを表現・変更・追跡できることを目的とする。

## Target Users

- **API 利用者**: Core-API（REST）で定義・ワークフロー実行を組み込む開発者・バックエンド担当
- **運用・開発者**: インシデント調査、ログ・メトリクス、定義変更の影響確認
- **UI 利用者**: Web ダッシュボード（Next.js）でグラフ・実行状況を視覚的に扱う担当

## Product Principles

- **契約の正本**: HTTP の形・エラー形式・主要ヘッダは `docs/` の契約ドキュメントと Core-API の実装を揃える（破壊的変更は意図的にのみ）。
- **レイヤー分離**: HTTP の入出力、ユースケース、永続化、エンジン実行の責務を混在させない（詳細は `structure.md`）。
- **機微データ（IO-14）**: `workflowInput` や state の `output` は一覧／GET でデフォルト露出しない方針を守る。デバッグ用に載る場合も外部ログ・画面に出す前にマスキング・サイズ制御を前提とする。
- **Core-API**: **C#（`api/`）のみ** を正とする。

## Success Metrics（プロダクト観点の例）

- 定義のデプロイから実行・キャンセルまでが API 契約どおりに再現できること
- 運用で **trace / tenant / 表示 ID** などでリクエストとドメインオブジェクトを追えること
- 契約・スキーマ・マイグレーションの変更が追跡可能であること

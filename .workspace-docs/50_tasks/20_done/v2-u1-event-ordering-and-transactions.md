# ハイブリッド方式での順序保証とトランザクション（U1 議論）

**前提**: Engine のイベント公開を **(1) コマンド戻り値** と **(2) コールバック** のハイブリッドで行う。

- **(1) コマンド直後**: Start / CancelAsync / PublishEvent(workflowId, eventName) の戻り値で `EventEnvelope[]` を返す。
- **(2) 非同期実行**: RunWorkflowAsync 内で発生するイベント（NODE*\* / JOIN*\* / EXECUTION_COMPLETED 等）はコールバックで API に渡す。

このドキュメントでは **順序保証** と **トランザクション境界** を議論する。

---

## 1. イベントの流れの整理

### 1.1 コマンドとイベントの対応

| コマンド                     | 戻り値で返すイベント（同期）                                                   | コールバックで返すイベント（非同期）                                                   |
| ---------------------------- | ------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------- |
| **Start**                    | EXECUTION_CREATED, EXECUTION_STARTED, （必要なら初期 NODE_CREATED/NODE_READY） | 各ノードの NODE*\* / JOIN*\* / 最終的に EXECUTION_COMPLETED または EXECUTION_FAILED 等 |
| **CancelAsync**              | EXECUTION_CANCEL_REQUESTED, （即時反映なら EXECUTION_CANCELED）                | 進行中ノードの NODE_CANCELED 等（必要に応じて）                                        |
| **PublishEvent(wfId, name)** | NODE_RESUME_REQUESTED 等                                                       | NODE\*RESUMED 以降の NODE\*_ / JOIN\_ 等                                               |

### 1.2 非同期側の並行性（現行 Engine）

- `ProcessFact` で **Fork** の場合、複数の `ScheduleStateAsync` を **await せず** fire-and-forget している（`_ = ScheduleStateAsync(...)`）。
- 複数ブランチが同時に進行し、**別々のタスク／スレッド**で「ノード完了」が起きうる。
- したがって「コールバック」は **同一 workflow に対して複数スレッドからほぼ同時に呼ばれうる**。

このため、**「発生順」の全順序**をどう決めるかが順序保証の中心になる。

---

## 2. seq の責務と付与タイミング

- **seq**: workflow_id 単位で単調増加する連番。event_store のリプレイは `WHERE workflow_id = ? ORDER BY seq` で行う。
- **誰が seq を決めるか**の選択肢:
  - **A) Engine が付与**: イベントを「出す」時点で Engine が seq を振る。API はそのまま INSERT。順序は「Engine が渡す順」に依存。
  - **B) API（DB）が付与**: INSERT 時に DB の sequence や `(SELECT MAX(seq)+1 ...)` で付与。順序は「API が INSERT する順」に依存。

・**議論ポイント**

- A の場合、**コールバックが複数スレッドから来る**と、Engine 側で「渡す順」を保証しない限り seq の大小が論理順とずれうる。
- B の場合、**API が「受け取った順」に INSERT する**必要があるが、複数スレッドから同時に受け取ると「受け取った順」が一意でない。  
  → **どちらにしても「1 workflow あたり 1 本の直列なストリーム」をどこかで作る必要がある。**

---

## 3. 順序保証の選択肢

### 3.1 案 A: Engine 側で「1 workflow あたり直列化」

- **仕様**: 同一 workflow に対して、**コールバックは常に 1 スレッドずつ**呼ばれる。かつ、**発生順**で呼ばれる。
- **実装イメージ**:
  - Engine 内に workflowId → **キュー** を用意。
  - 状態変化が起きるたびに「イベント」をキューに enqueue。
  - 1 本の **処理ループ**（または lock 内で dequeue）がキューから取り出し、**順に** コールバックを 1 回ずつ呼ぶ。
- **seq**:
  - **Option A1**: Engine がキューに入れる時点で seq を振る（workflow 内で単調増加）。API はそのまま INSERT。
  - **Option A2**: seq は API が INSERT 時に付与。Engine は順序だけ保証し、API は「受け取った順」に INSERT するので seq は自然に単調増加。
- **メリット**: API は「コールバックが順に来る」と仮定でき、1 件ずつ INSERT + reducer + projection 更新をすればよい。トランザクションが単純。
- **デメリット**: Engine の実装が重い（キュー・ループ・スレッドモデル）。また、Fork で「どちらのブランチを先にキューに入れるか」を決める必要がある（実装の詳細で順序が決まる）。

### 3.2 案 B: API 側で「1 workflow あたり 1 キューで直列処理」

- **仕様**: Engine のコールバックは **複数スレッドから呼ばれてよい**。各コールバックでは「イベント（1 件または複数）」を渡すだけ。  
  API 側で **workflowId ごとに 1 本のキュー** を持ち、コールバックで受け取ったイベントをキューに enqueue。  
  ワーカーがキューから **1 件ずつ（または 1 バッチずつ）** 取り出し、**その順** で event_store INSERT + reducer + projection 更新を行う。
- **問題**: コールバックが「複数スレッドから同時」に来ると、**キューに入る順序**が「論理的な発生順」と一致するか。
  - 一致させるには、**Engine がイベントに「論理順」を付与**する必要がある（例: 単調増加する sequence を Engine が振る。API はそれでソートしてからキューに入れるか、または Engine が順序付きで渡す）。
  - あるいは **Engine は順序を保証せず**、イベントに **causationId / 親子関係** だけ付け、API は「依存が解消した順」に処理する（DAG のトポロジカルソート）。これは実装が重い。
- **シンプルな形**: **Engine が各イベントに「workflow 内の論理 seq」を付与**する（Engine 側で 1 カウンタでよい）。API は workflowId 単位でキューに入れ、**その seq でソート**してから順に INSERT。これなら「API 側 1 キュー」でも順序は保証できる。  
  → 実質「seq の付与は Engine、永続化時の seq は Engine のそれを使う」に落ちる。

### 3.3 案 C: コールバックは「バッチ」で 1 回だけ、Engine が順序付きで渡す

- **仕様**: Engine は **同一 workflow のイベントを、内部で直列化してから**、**「ここまでで発生したイベント列」を 1 リストで**コールバックに渡す（案 A に近いが、1 回のコールバックで複数イベント）。  
  つまり「キューがたまったところで」「またはノード 1 つ完了するたびに」、**それまでに確定した順序付きイベント列**を 1 回で渡す。
- **seq**: Engine がリスト内で seq を振るか、API が INSERT 順で振る（リストの順がそのまま seq 順）。
- **メリット**: API は「1 回のコールバック = 順序保証された 1 バッチ」として 1 トランザクションで INSERT + reducer + projection 更新ができる。
- **デメリット**: Engine 側で「いつバッチを切るか」「直列化をどう実装するか」の設計が必要。

## 4. トランザクション境界の整理

### 4.1 (1) コマンド戻り値で返すイベント

- **流れ**: API が Engine を呼ぶ → Engine が `EventEnvelope[]` を返す → API が 1 トランザクションで:
  1. event_store に **まとめて INSERT**（seq は 1..n または「前回の最大 seq + 1」から）
  2. reducer を **そのイベント列に対して** 適用（必要なら既存の workflow 状態とマージ）
  3. workflows / execution_graph_snapshots を **更新**
- **ポイント**: 1 コマンド = 1 トランザクション。順序は「戻り値の配列の順」で確定。

### 4.2 (2) コールバックで渡すイベント

- **案 A を採用した場合（Engine が 1 件ずつ順にコールバック）**
  - 1 回のコールバック = 1 イベント。
  - API の 1 トランザクション: **1 件 INSERT** → reducer(現在の projection, その 1 件) → **projection 更新**。
  - これで seq の単調増加と projection の一貫性が保たれる。

- **案 C を採用した場合（Engine が順序付きバッチでコールバック）**
  - 1 回のコールバック = 順序付きの `EventEnvelope[]`。
  - API の 1 トランザクション: **バッチをまとめて INSERT**（seq は配列順）→ reducer を **バッチ全体** に適用 → **projection 更新**。
  - 1 バッチ = 1 トランザクションなので、途中失敗時は「バッチ全体をロールバック」でよい。

### 4.3 コマンド直後とコールバックの seq のつなぎ

- **Start** の戻り値で [EXECUTION_CREATED, EXECUTION_STARTED] を返す → API が seq 1, 2 で INSERT。
- 続くコールバックで [NODE_CREATED, NODE_READY, ...] が来る → API は **seq 3 から** INSERT。
- 実装方法:
  - **Engine が seq を振る場合**: コマンド戻り値のイベントには seq を含めない（API が INSERT 時に 1, 2 を付与）。コールバック側のイベントには Engine が「3 以降」を振るためには、Engine が「その workflow の現在の最大 seq」を知る必要がある → **API がコマンド処理後に「次は seq 3」を Engine に渡す**か、**Engine は seq を振らず、API が常に INSERT 時に MAX(seq)+1 で付与**する。
  - **API が seq を振る場合**: コマンド処理トランザクションで seq 1, 2 を付与。コールバック処理では「その workflow の現在の最大 seq」を取得してから INSERT（SELECT FOR UPDATE や sequence で次を取得）。これなら Engine は seq を知らなくてよい。

## 5. 失敗・リトライ

- **コマンド側**: API が「戻り値のイベントを INSERT + projection 更新」でトランザクション失敗した場合、**コマンド全体をリトライ**可能（Engine の Start を再度呼ぶと別 workflowId になるので、元の Start は「永続化されなかった」として扱い、クライアントに 5xx を返してリトライさせる）。
- **コールバック側**:
  - **1 件ずつ処理**: 1 件の INSERT + reducer + projection が失敗したら、その 1 件をリトライ。キューに戻すか、死信キューに退避するか。
  - **バッチ処理**: バッチ全体が失敗したらバッチをリトライ。Engine が「同じバッチを再送」できる必要がある（べき等にすると、同じイベントを再度コールバックで送り、API が eventId で重複排除する、など）。

---

## 6. 推奨の方向性（議論用）

1. **順序保証**: **案 A（Engine 側で 1 workflow あたり直列化）** または **案 C（Engine が順序付きバッチでコールバック）** のどちらかにすると、API の責務が単純になる。
   - 案 A: コールバックは 1 件ずつ、順番に来る。
   - 案 C: コールバックは「順序付きのリスト」で来る。

2. **seq**: **API（DB）が INSERT 時に付与** にすると、Engine は「順序」だけ保証すればよく、seq の値は DB に任せられる。
   - コマンド戻り値: 同じトランザクションで INSERT するので、MAX(seq)+1 から順に付与。
   - コールバック: 「順に 1 件ずつ」または「バッチ内の順序」で INSERT するので、同様に MAX(seq)+1 で付与。

3. **トランザクション**:
   - コマンド: 戻り値の Event[] を **1 トランザクション** で INSERT + reducer + projection。
   - コールバック（案 A）: **1 イベント = 1 トランザクション**（INSERT + reducer + projection）。
   - コールバック（案 C）: **1 バッチ = 1 トランザクション**（INSERT 複数 + reducer + projection）。

4. **失敗時**: コールバック処理が失敗した場合、**Engine 側で同じイベントを再送できる**ようにするか、**API 側でキューに保持してリトライ**するかは、案 A/C の実装（キューを Engine が持つか API が持つか）に依存する。

---

## 7. 決定事項・オープンな論点

### 7.1 決定事項

| 事項                       | 決定内容                                                                                                                                                            |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **順序保証の方式**         | **案 C** を採用。Engine が同一 workflow のイベントを内部で直列化し、**順序付きバッチ**でコールバックに渡す。1 回のコールバック = 順序保証された `EventEnvelope[]`。 |
| **Fork 時の論理順**        | **スケジューラの実装に準じる**。複数ブランチが並行して進むときのリプレイ順序は、Engine のスケジューラがイベントをキューに載せる順序とする。                         |
| **監査用 workflow_events** | event_store への INSERT と **同一トランザクション**で workflow_events にも INSERT する。                                                                            |

### 7.2 オープンな論点

- **reducer の所在**: Engine が持つか API が持つかは **open-decisions の U7 で再検討**する。

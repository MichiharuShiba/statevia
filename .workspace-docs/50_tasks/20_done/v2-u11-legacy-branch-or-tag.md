# U11: legacy 保存はブランチかタグか

**前提**（modification-plan Phase 3, open-decisions 5.1）: **TypeScript Core-API**（`services/core-api/`）は v2 では使用せず、**削除する**。削除前に **Git で履歴を残す** ため、「legacy ブランチ」または「legacy タグ」のいずれかで保存する、と計画に書かれている。

本ドキュメントでは **ブランチとタグのどちらを採用するか** と、**運用（マージ方針・履歴の残し方）** を議論する。

---

## 1. 論点

- **保存の単位**: TypeScript Core-API が存在する**最後のコミット**を指す名前を付ける。その名前を **ブランチ** にするか **タグ** にするか。
- **運用**: ブランチの場合は「マージしない・削除しない」方針を明文化するか。タグの場合は「読み取り専用のスナップショット」として扱う。
- **名前**: 例として `legacy/core-api-ts`（ブランチ）または `legacy/core-api-ts-v1`（タグ）など、意図が分かる名前を決める。

---

## 2. 選択肢

| 案    | 方法         | 手順例                                                                                                            | メリット                                                                 | デメリット                                                                         |
| ----- | ------------ | ----------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ | ---------------------------------------------------------------------------------- |
| **A** | **ブランチ** | 削除前に `git checkout -b legacy/core-api-ts` でブランチを作り push。その後 main で `services/core-api/` を削除。 | 必要ならそのブランチ上で参照・比較しやすい。                             | ブランチは「開発の延長」と誤解され、マージや削除のリスクがある。名前の運用が続く。 |
| **B** | **タグ**     | 削除前のコミットに `git tag legacy/core-api-ts-v1` を打ち push。その後 main で削除。                              | **不変の参照**。リリース・スナップショットの意図が明確。マージされない。 | タグは「リリース版」と混同されうる。ブランチより参照頻度は低い運用になりがち。     |

**補足**: Git の慣習では **タグ** は「ある時点のスナップショットを名前で指す」用途に使うことが多く、**ブランチ** は「開発の流れ」に使う。legacy 保存は「削除前の状態を残すだけ」なので、**タグの方が意図に合う**。ブランチを選ぶ場合は「このブランチにはマージしない・main から分岐した読み取り用」と README や運用ルールに明記する必要がある。

---

## 3. 推奨の方向性

- **採用**: **タグ**（案 B）を推奨する。削除する直前のコミット（TypeScript Core-API が含まれる最後のコミット）に **annotated タグ** を打ち、`legacy/core-api-ts` または `legacy/core-api-ts-v1` のような名前で push する。
- **手順**: (1) Phase 3 で `services/core-api/` を削除するコミットの**直前**のコミットを特定する。(2) そのコミットに `git tag -a legacy/core-api-ts -m "Last state of TypeScript Core-API before v2 removal"` を実行。(3) `git push origin legacy/core-api-ts`。(4) その後、main 上で `services/core-api/` を削除するコミットを入れる。
- **ドキュメント**: README または AGENTS.md に「TypeScript Core-API の最終状態はタグ `legacy/core-api-ts` で参照できる」と記載する。

---

## 4. 決定事項・オープンな論点（記入用）

### 4.1 決定事項

| 事項         | 決定内容                                                                                     |
| ------------ | -------------------------------------------------------------------------------------------- |
| 保存方法     | **タグ** を採用する（決定）                                                                  |
| タグ名       | **legacy/core-api-ts**                                                                       |
| 手順         | TypeScript Core-API 削除**直前**のコミットに annotated タグを打ち push。その後 main で削除。 |
| ドキュメント | README / AGENTS.md に「最終状態はタグ `legacy/core-api-ts` で参照可能」と明記（決定）        |

### 4.2 オープンな論点

- タグ名を `legacy/core-api-ts` に統一するか、日付やバージョンを含めるか（例: `legacy/core-api-ts-2025-03`）。
- 複数リポジトリやフォークがある場合、同じタグ名を同じ意味で使うか。

以上を、U11 legacy 保存はブランチかタグかの議論とする。

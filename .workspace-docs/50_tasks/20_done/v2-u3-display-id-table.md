# U3: 表示用 ID の専用テーブル議論

**前提**（open-decisions 9.1 / 9.2）: definitionId および workflowId は **UUID** を PK とする。**表示用 ID（英数字 10 桁）** を別途持ち、専用テーブルで UUID と対応付け、**表示用 ID から UUID に変換**して各テーブルにアクセスする。表示用 ID は PK にしない。

このドキュメントでは **専用テーブルの構成**（1 テーブルか 2 テーブルか、テーブル名・列名）と **表示用 ID の生成規則** を議論する。

---

## 1. 表示用 ID の役割

| 用途     | 説明                                                                          |
| -------- | ----------------------------------------------------------------------------- |
| **URL**  | `/v1/definitions/abc12xyz`, `/v1/workflows/def34uvw` のように短いパスにする。 |
| **表示** | 一覧・詳細画面でユーザーに示す。コピー・口頭での共有をしやすくする。          |
| **ログ** | 長い UUID の代わりに短い ID を出力して可読性を上げる。                        |

**制約**: 英数字 10 桁（桁数は決定済み）。衝突時は **DB の UNIQUE 制約で弾き、再生成**する運用を前提とする（open-decisions 9 補足）。

---

## 2. テーブル構成の選択肢

### 2.1 案 A: 1 テーブルで definition / workflow の両方を管理

**テーブル名の例**: `display_id_mapping`, `id_aliases`, `short_ids` など。

#### 案 A の列イメージ

| column     | type        | 説明                                                               |
| ---------- | ----------- | ------------------------------------------------------------------ |
| kind       | varchar     | `"definition"` または `"workflow"`。どのエンティティの短縮 ID か。 |
| display_id | varchar(10) | 英数字 10 桁。UNIQUE 制約（全体で 1 件だけ）。                    |
| uuid       | uuid        | 対応する definition_id または workflow_id。                        |
| created_at | timestamptz | 作成時刻（任意）。                                                 |

#### 案 A の主キー・UNIQUE

- **主キー**: (kind, uuid) または (display_id)。
- **UNIQUE**: display_id（kind に関係なくグローバルで一意）。

#### 案 A のメリット

- テーブルが 1 つで済む。マイグレーション・運用が単純。
- 表示用 ID の名前空間が 1 つなので、「10 桁で definition と workflow が被らない」ようにできる（被っても kind で区別可能だが、URL 上は `/definitions/xxx` と `/workflows/xxx` でパスが分かれるので、同じ display_id が definition と workflow に同時に存在すると混乱する可能性はある → **display_id はグローバルで一意**にすれば問題ない）。

#### 案 A のデメリット

- kind で区切るため、「definition の display_id 一覧」などは `WHERE kind = 'definition'` が必要。
- definition 用と workflow 用で列やインデックスを分けたい場合に融通が利きにくい（通常は同じ列で足りる）。

---

### 2.2 案 B: 2 テーブルに分ける（definition 用 / workflow 用）

**テーブル名の例**: `definition_display_ids`, `workflow_display_ids`。または `definitions_short_ids`, `workflows_short_ids`。

#### 案 B の列イメージ（各テーブルとも）

| column        | type        | 説明                                           |
| ------------- | ----------- | ---------------------------------------------- |
| display_id    | varchar(10) | 英数字 10 桁。UNIQUE。                         |
| definition_id | uuid        | 対応する UUID。（definition 用テーブルの場合） |
| workflow_id   | uuid        | 対応する UUID。（workflow 用テーブルの場合）   |
| created_at    | timestamptz | 任意。                                         |

#### 案 B のメリット

- エンティティごとにスキーマが明確。definition にだけ列を足す、などがしやすい。
- クエリで「definition の display_id」だけを扱うとき、テーブルが分かれていると意図が明確。

#### 案 B のデメリット

- テーブルが 2 つ。マイグレーション・コードの参照が 2 箇所になる。
- 表示用 ID の生成ロジックは 2 テーブルで共通化する必要がある。

---

### 2.3 案 C: 各親テーブルに display_id 列を追加（専用テーブルを持たない）

workflow_definitions に `display_id varchar(10) UNIQUE`、workflows に `display_id varchar(10) UNIQUE` を追加する。

#### 案 C のメリット

- 専用テーブルが不要。INSERT 時に display_id を生成して同じ行に書くだけ。
- 「表示用 ID → UUID」の変換は、該当テーブルを SELECT するだけ。

#### 案 C のデメリット

- 計画では「専用テーブルで管理」としているため、この案は **前提と合わない**。
- 参考として「専用テーブルをやめて親テーブルに持たせる」選択肢があることだけ記載する。**採用する場合は open-decisions 9.1 / 9.2 の「専用テーブルで管理」を見直す必要がある。**

---

## 3. 表示用 ID（10 桁英数字）の生成規則

桁数は **10 桁**、文字種は **英数字**（a–z, A–Z, 0–9 の 62 文字、または 0–9 + a–z の 36 文字など）を想定。

### 生成方法の候補

| 方式              | 説明                                                                    | 衝突リスク                                                           | 分散対応                                          | 備考                                                                                                                                                                                                                                                                                       |
| ----------------- | ----------------------------------------------------------------------- | -------------------------------------------------------------------- | ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **乱数**          | 10 桁をランダムに生成。                                                  | 62^8 ≈ 2e14 通り。衝突確率は件数が増えると上がる。                   | 複数 API インスタンスでも衝突しうる。             | **UNIQUE 制約で重複を弾き、挿入失敗時に再生成**で対応。                                                                                                                                                                                                                                    |
| **DB シーケンス** | シーケンスの nextval を 10 桁英数字にエンコード（例: 36 進数）。         | 同一 DB なら一意。                                                   | 単一 DB 前提。複数 API は同じ DB を参照すれば可。 | 順序が推測されやすい。桁あふれは 36^8 まで。                                                                                                                                                                                                                                               |
| **時刻ベース**    | ナノ秒やミリ秒 + 乱数で 10 桁に圧縮。                                    | 同一瞬間の並行で衝突しうる。                                         | 複数インスタンスで時刻ずれがあると衝突リスク。    | UNIQUE + 再生成が現実的。                                                                                                                                                                                                                                                                  |
| **UUID の短縮**   | UUID の一部（例: 先頭 5 バイトを 16 進で 10 文字）を display_id にする。 | 16^10 通り。UUID が一意なら理論上被らないが、10 桁に切ると衝突しうる。 | UUID は分散で一意なので、短縮のみが衝突要因。     | **元の UUID と 1:1 になる**が、10 桁だけでは一意にならないため、別途「10 桁 → UUID」のマッピングが必要。同じ UUID からは同じ 10 桁になるので、表示用としては「一意な短縮表示」ではなく「短い表示」になる。一意性を保つなら UNIQUE 制約で被りを弾くか、衝突時に別の 10 桁を生成する必要がある。 |

### 推奨の方向性（議論用）

- **乱数（英数字 10 桁）** + **INSERT 時の UNIQUE 制約**。衝突したらアプリで再生成してリトライ。実装が単純で、複数 API インスタンスでも同じ DB を参照すれば一意性は DB が担保する。
- 採番の順序や推測可能性を気にしないなら **DB シーケンスを 10 桁にエンコード** も可。単一 DB で十分な場合に向く。

## 3.1 62 文字種・10 桁の衝突確率と「現実的な有効数」

- **総数**: \(N = 62^8 = 218{,}340{,}105{,}584{,}896\)（約 2.18 × 10¹⁴、約 218 兆通り）。

### （A）既に k 件あるときに、新規 1 件を生成して衝突する確率

- 近似: \(P \approx k / N\)（k ≪ N のとき）。
- 目安:
  - **100 万件**（10⁶）: \(P \approx 4.6 \times 10^{-9}\)（約 0.0000005%）→ 実質無視できる。
  - **1,000 万件**（10⁷）: \(P \approx 4.6 \times 10^{-8}\)（約 0.000005%）→ 非常に低い。
  - **1 億件**（10⁸）: \(P \approx 4.6 \times 10^{-7}\)（約 0.00005%）→ 依然として低い。
  - **10 億件**（10⁹）: \(P \approx 4.6 \times 10^{-6}\)（約 0.0005%）→ まだ低いが、UNIQUE + 再生成を前提とする。
  - **100 億件**（10¹⁰）: \(P \approx 4.6 \times 10^{-5}\)（約 0.005%）→ 衝突は稀だが、リトライは想定する。

### （B）これから k 件をランダムに生成したとき、その中で 1 回以上衝突が起きる確率（誕生日問題）

- 近似: \(P \approx 1 - \exp\bigl(-k^2 / (2N)\bigr)\)。
- 50% になる k: \(k \approx \sqrt{2 N \ln 2} \approx 1.74 \times 10^7\)（約 **1,740 万件**）。  
  → 約 1,740 万件まで「無衝突で増やせる」期待が約半分。それ以上は「どこかで衝突が起きる」可能性が 50% 超。

### 現実的な「衝突が起きにくい」有効数の目安（10 桁）

| 許容したい「新規 1 件の衝突確率」 | 既存 ID 数の目安（k） | 備考                      |
| --------------------------------- | --------------------- | ------------------------- |
| 0.0001% 未満                      | 約 2,180 万件未満     | 新規 1 件あたり非常に安全 |
| 0.001% 未満                       | 約 2.18 億件未満      | 多くのサービスで十分安全  |
| 0.01% 未満                        | 約 21.8 億件未満      | 非常に多い件数まで許容    |
| 1% 未満                           | 約 2.18 兆件未満      | 理論上ほぼ上限に近い      |

### 結論（10 桁の運用の目安）

- **数百万〜数千万件**まで: 衝突は極めて稀。UNIQUE + 再生成で十分現実的。
- **数億件**まで: 新規 1 件の衝突確率はまだ低い。同じく UNIQUE + 再生成で問題ない。
- **数十億件**を超える規模では、衝突確率が 0.01% を超えていくので、リトライ回数や監視を少し強くする想定がよい。
- 62 文字・10 桁であれば、**実用上は「数十億件」を「衝突リスクが低い」有効数の目安**にできる（8 桁の約 62 倍）。それ以上でも UNIQUE 制約と再生成で運用可能。

## 3.2 62 文字種・10 桁の論理数と有効数

- **論理数（総数）**: \(N = 62^{10} = 839{,}299{,}365{,}868{,}340{,}224\)（約 8.39 × 10¹⁷、約 **839 京**通り）。8 桁の約 3,844 倍。

### （A）既に k 件あるときに、新規 1 件を生成して衝突する確率（10 桁）

- 近似: \(P \approx k / N\)（k ≪ N のとき）。
- 目安:
  - **100 万件**（10⁶）: \(P \approx 1.2 \times 10^{-12}\) → 実質無視できる。
  - **1,000 万件**（10⁷）: \(P \approx 1.2 \times 10^{-11}\) → 極めて低い。
  - **1 億件**（10⁸）: \(P \approx 1.2 \times 10^{-10}\) → 非常に低い。
  - **10 億件**（10⁹）: \(P \approx 1.2 \times 10^{-9}\)（約 0.00000012%）→ 依然として低い。
  - **100 億件**（10¹⁰）: \(P \approx 1.2 \times 10^{-8}\)（約 0.0000012%）→ まだ非常に低い。
  - **1,000 億件**（10¹¹）: \(P \approx 1.2 \times 10^{-7}\)（約 0.000012%）→ 低い。UNIQUE + 再生成で十分。

### （B）これから k 件をランダムに生成したとき、その中で 1 回以上衝突が起きる確率（誕生日問題・10 桁）

- 近似: \(P \approx 1 - \exp\bigl(-k^2 / (2N)\bigr)\)。
- 50% になる k: \(k \approx \sqrt{2 N \ln 2} \approx 1.08 \times 10^9\)（約 **10.8 億件**）。  
  → 約 10.8 億件まで「無衝突で増やせる」期待が約半分。8 桁より約 62 倍多い。

### 現実的な「衝突が起きにくい」有効数の目安（10 桁）

| 許容したい「新規 1 件の衝突確率」 | 既存 ID 数の目安（k） | 備考                      |
| --------------------------------- | --------------------- | ------------------------- |
| 0.0001% 未満                      | 約 8,390 億件未満     | 新規 1 件あたり非常に安全 |
| 0.001% 未満                       | 約 8.39 兆件未満      | 膨大な件数まで許容        |
| 0.01% 未満                        | 約 83.9 兆件未満      | 理論上ほぼ上限に近い      |

### 結論（10 桁の運用の目安）

- **数億〜数十億件**まで: 衝突は極めて稀。UNIQUE + 再生成で十分現実的。
- **数百億件**まで: 新規 1 件の衝突確率はまだ 0.0001% 未満。有効数として十分大きい。
- 62 文字・10 桁であれば、**実用上は「数十億件」を「衝突リスクが低い」有効数の目安**にできる。誕生日問題の 50% 点も約 10.8 億件と 8 桁より大幅に伸びる。

## 4. 衝突時の挙動

- **INSERT 時に display_id の UNIQUE 制約に違反**した場合 → アプリで **表示用 ID を再生成してリトライ**（最大 N 回など上限を設ける）。
- 再生成しても衝突し続ける場合は **5xx または 409** を返し、運用で対応する（極めて稀な想定）。
- 生成規則で「UUID の短縮」を選んだ場合、**同じ UUID からは同じ 10 桁**になるため、2 回目以降の INSERT では必ず衝突する。そのため「表示用 ID は UUID に依存しない乱数またはシーケンス」とする方がよい。

## 5. テーブル名・列名の候補

### 5.1 1 テーブル（案 A）の場合

| 候補 | テーブル名         | display_id 列 | UUID 列                       |
| ---- | ------------------ | ------------- | ----------------------------- |
| 1    | display_id_mapping | display_id    | uuid, kind                    |
| 2    | id_aliases         | short_id      | entity_kind, entity_id (uuid) |
| 3    | short_ids          | id (10 桁)    | kind, ref_uuid                |

**推奨**: `display_id_mapping`。列は `kind`, `display_id`, `uuid`, `created_at`。  
PK は (kind, uuid)。UNIQUE(display_id)。

### 5.2 2 テーブル（案 B）の場合

- `definition_display_ids`: display_id, definition_id (uuid), created_at
- `workflow_display_ids`: display_id, workflow_id (uuid), created_at

いずれも display_id に UNIQUE 制約。  
同じ display_id が definition と workflow の両方に存在することは許容するか否か（URL が `/definitions/abc12` と `/workflows/abc12` で別リソースなので、被っても識別は可能。運用方針で「被らせない」なら 1 テーブルでグローバル UNIQUE の方がよい）。

## 6. その他の検討

- **英数字の範囲**: 「0-9, a-z」の 36 文字にするか、「0-9, a-z, A-Z」の 62 文字にするか。URL で case-insensitive に扱うなら 36 文字で統一すると安全。
- **created_at**: 監査・デバッグ用にあるとよい。必須かどうかは運用次第。
- **U4 との関係**: GET の {id} が表示用 ID の場合、このテーブルで display_id → uuid を解決してから workflow_definitions / workflows を検索する。U4 で「URL の id は表示用 ID」と決まれば、このテーブルが必須になる。

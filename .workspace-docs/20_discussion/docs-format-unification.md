# docs format unification

更新日: 2026-04-02  
対象: `.workspace-docs/README.md`, `.workspace-docs/*/*`

## 背景

`.workspace-docs` 配下の文書は実用上十分に運用できている一方で、次の揺れが残っている。

- タスク表の列構成と記法が文書ごとに微妙に異なる
- フォルダ構成（`discussion` 追加後）の責務が README 上では最小説明に留まっている
- ファイル名の命名規則が明文化されておらず、将来の検索性と移動ルールが曖昧

この文書では、フォーマット統一に加えて「フォルダ名再議論」「ファイル名命名ルール」を同時に整理する。

## 論点

1. タスク系 Markdown の表フォーマットをどこまで固定するか
2. `.workspace-docs` 配下フォルダの責務と名称を現状維持するか、改名するか
3. ファイル名命名ルールをどの粒度まで強制するか
4. 合意後の反映手順（段階移行か一括移行か）をどうするか

## 選択肢

### A. 最小統一（軽量運用）

- タスク表のみ必須ルール化: `ID | 状態 | タスク | ...`
- フォルダ名は現状維持（`discussion` を含む）
- ファイル名は「既存尊重 + 新規のみルール適用」

長所:

- 既存文書の改修コストが小さい
- すぐに運用へ乗せられる

短所:

- 古い命名や構成の揺れがしばらく残る

### B. 中間統一（推奨）

- タスク表は必須ルール化
- `30_specs/40_plans/50_tasks` それぞれの章テンプレートを定義
- ファイル名命名ルールを新規・更新ファイルに適用
- フォルダ名は現状維持、必要なら「表示名（README見出し）」で補助

長所:

- 検索性と可読性が着実に上がる
- 一括リネームより安全

短所:

- 反映ルールの運用が必要

### C. 完全統一（厳格運用）

- 全文書を一括で命名・章構成・表形式を再整備
- フォルダ名見直しも同時実施（必要なら移動とリンク更新を伴う）

長所:

- ルールが最も明確

短所:

- 変更量が大きく、参照切れやレビュー負荷のリスクがある

## フォルダ名の再議論（案）

現状:

- `inbox`, `notes`, `discussion`, `specs`, `plans`, `tasks`

論点:

- `notes` と `discussion` の棲み分けを明確化するか
- `inbox` を維持するか（週次で空にできる運用があるか）

暫定案:

- フォルダ名は当面維持（改名はしない）
- 役割のみ明文化して重複を防ぐ
  - `discussion`: 合意形成プロセス
  - `notes`: 個人メモ・調査記録・草案

## ファイル名命名ルール（決定前メモ）

基本方針:

- 小文字 kebab-case を原則とする
- 先頭にドメイン接頭辞を付与して検索しやすくする
- 状態（in-progress/done/archived）はファイル名に含めない

検討していたパターン:

- `specs`: `v2-<topic>-spec.md`
- `plans`: `v2-<topic>-plan.md`
- `tasks`: `v2-<topic>-tasks.md` または `v2-<topic>-backlog.md`
- `discussion`: `<topic>-discussion.md` または `<topic>-decision-log.md`

禁止・非推奨:

- 空白文字を含むファイル名
- 同義語の混在（例: `todo`, `task`, `backlog` の無秩序併用）
- 日付のみで意味が分からない名前

## 決定（2026-04-01 〜 2026-04-02）

### タスク表フォーマット（固定）

以下を標準フォーマットとして固定する。

```md
| ID | 状態 | タスク | 優先度 | 依存 | 完了条件 | 備考 |
|----|------|--------|--------|------|----------|------|
```

### 記述ルール（採用）

- 状態は必ず `状態` 列で管理する（`ID` や `タスク` に「（完了）」を書かない）
- 並び順は原則 `ID` 昇順。実行順を優先する場合は `備考` に「実装順」を明記する
- 空欄は禁止。未設定値は `-` を使う
- 表の直前に対象スコープを 1 行で記載する（例: `対象: v2 Core-API`）

### 運用ルール（採用）

- 行追加時は `ID` 重複を確認する
- 状態変更時は関連する親表（例: `v2-remaining-tasks.md`）も同時更新する
- 詳細は表に詰め込まず、必要に応じて下段の節へ分離する

### フォルダ責務（採用）

- `20_discussion`: 開発者と AI の対話・合意形成の場として利用する
- `10_notes`: 開発者の思想・個人メモ・調査記録の場として利用する
- `00_inbox`: 現時点では残す。運用実績を見て将来再検討する

### ファイル名命名ルール（採用）

- 既存ファイル名は現状維持（リネームしない）
- 次回以降の新規ファイルは `v2-` などのバージョン接頭辞を付けない
- 次回以降の新規ファイルは **`{何を}_{どうする}_{種別}.md`** とする
  - 3セグメントの区切りは **アンダースコア `_`**
  - 各セグメント内の語の区切りは **ハイフン `-`**（小文字 kebab-case）
- 状態（`10_in-progress` / `20_done` / `30_archived`）はファイル名に含めない
- バージョン情報はファイル名ではなく本文で管理する方針とする（後述のメタブロック）

### 本文メタブロック（採用・新規ファイル）

- タイトル（`# ...`）の **直後** に、次の **箇条書きメタブロック** を置く（空行 1 行のあと `---` で本文と区切る）。
- **Version**（必須）: [セマンティックバージョン](https://semver.org/lang/ja/)（`MAJOR.MINOR.PATCH` のみ。先頭に `v` は付けない）。
  - **現時点（プロダクト未リリース）**: 新規文書は **`1.0.0` から開始**する（複数文書で揃える）。
  - **プロダクトリリース後**: 当該文書の内容変更に応じて **必ずバージョンを増やす**（増分の MINOR / PATCH / MAJOR の厳密ルールは運用しながら必要に応じて細則化する）。
- **更新日**（必須）: `YYYY-MM-DD`。
- **対象**（推奨）: スコープの一行要約（例: `Core-API`, `Engine`, `.workspace-docs 運用`）。
- **関連**（任意）: 参照するほか文書への相対パスを列挙。
- **Status は書かない**: 文書のライフサイクルは **`10_in-progress` / `20_done` / `30_archived` のフォルダ**で表す。タスク系の「行ごとの状態」は **表の `状態` 列**でのみ管理する。

**既存ファイル**: 即時の全面改修はしない。次回そのファイルを**本質的に更新するとき**にメタブロックへ寄せる。

### 種別（固定語彙・採用）

次の **5語のみ** を種別として用いる（提案A）。運用の中で不足があれば **提案B**（`decision` / `memo` など）を別途検討する。

| 種別 | 意味 | 主な置き場 |
|------|------|------------|
| `spec` | 要件・契約・設計の正本 | `30_specs` |
| `plan` | 方針・段階・マイルストーン・意思決定の整理 | `40_plans` |
| `tasks` | 実装・検証のチェック可能なタスク表 | `50_tasks` |
| `backlog` | 将来・見送りの積み残し（まだ実装単位に割れないもの） | **`50_tasks` のみ** |
| `discussion` | 合意前の論点・選択肢・決定待ち | `20_discussion` |

**`tasks` と `backlog` の目安**: 着手単位がスプリントで切れるなら `tasks`、まだ粒度が粗い・時期未定なら `backlog`。

**`backlog` の配置**: 当面は `50_tasks` にのみ置く。運用上つらくなったら **専用フォルダの新設**を検討する。

## 保留

- 既存ファイルの一括リネームを行うか
- `10_notes` と `20_discussion` 間の再配置基準（どの時点で移すか）
- プロダクトリリース後の **Version 増分細則**（変更規模と MINOR / PATCH / MAJOR の対応）
- 種別の拡張（提案B: `decision` / `memo` / `readme` 相当の扱い）

## 正本への反映先

- `.workspace-docs/README.md`（配置ルール・命名ルールの明文化）
- `.workspace-docs/50_tasks/10_in-progress/*`（表フォーマット最終統一）
- 必要に応じて `.workspace-docs/40_plans/10_in-progress/*`, `.workspace-docs/30_specs/10_in-progress/*`
